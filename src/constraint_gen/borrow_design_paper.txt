## Design

typedef string cpath_part; 
typedef vector<cpath_part> cpath;
// The first part of a cpath is a variable name or "$func_name::return" or "$func_name::args::$arg_name".
// Each additional part of a cpath is either:
//  - "[0]" (for pointer deref or array indexing) or
//  - ".fieldname" for struct fields.
// We sometimes use the term "base var" to refer to the first part of a cpath,
// and we say that the C-path is 'rooted' at the base var.
//
// IMPORTANT: Unlike C, given a pointer p to a struct, we write "p.field1" to
// denote the address of the field in struct pointed to by p, rather than
// "&(*p).field".  We never write "p[0].field" to denote the value of field;
// instead, we write "p.field[0]" for that, in order to align with the LLVM
// GEP instruction.

// Note that a good responsible pointer cpath becomes a zombie under two cases:
// (1) the pointer is freed or (2) the pointer is transferred to another
// responsible cpath.  We generate constraints below to prohibit a responsible
// pointer from becoming a zombie when there are still outstanding borrows from it.

// Note that, unlike Rust, we do not have `mut` and `responsible` annotations
// on everything from the get-go (the `.pom.yml` specifies them for args,
// locals, and returns, but not for results of subexpressions); thus we
// generate constraints that are valid regardless.  This is why the BorrowInfo
// data doesn't record which cpaths are mut and/or responsible -- we don't know
// it yet.

// We do not support unions or structs that contain pointers.
// We currently do not support recursive data structures such as linked lists.
// The main things that are missing (besides properly handing structs at all)
// are (1) generating constraints and borrow/lifetime info when a pointer to a
// struct is loaded from memory and
// (2) verifying that invariants (e.g., no pointers in the struct are zombies)
// aren’t violated except briefly (e.g., verifying that invariants are always
// satisfied before function calls and at the end of basic blocks),
// (3) dealing with args and returns whose type is a struct type in C (they
// get changed during IR gen in Clang).

// We catch pointers to local vars that escape that function, but we currently
// do not support catching pointers to locals inside an interior scope that
// escape the scope of the local but not the scope of the function.
// This is because we operate at the LLVM IR level, and conversion to LLVM IR
// does not preserve this sort of use-after-free error.

// Rust allows different fields of a struct to be borrowed independently.  We
// currently don't support this; borrowing a field of a struct borrows the
// whole struct.  Allowing fields to be borrowed independently will be future work.


## Transitive borrows

We write "borrow+" as an abbreviation of "transitive borrow".
For MAY_OR_MUST in [MAY, MUST]:
 - If there is a direct borrow from p1 to p2, then there is a MAY_OR_MUST borrow+ from p1 to p2.
 - If there is a MAY_OR_MUST borrow+ from p1 to p2 and a MAY_OR_MUST borrow+ from p2 to p3, then there is a MAY_OR_MUST borrow+ from p1 to p3.
 - If there is a MAY_OR_MUST borrow+ from p1 to p2, and ptr_depth(p1) >= 2 and ptr_depth(p2) >= 2, then there is a MAY_OR_MUST borrow+ from *p1 to *p2.
 - If there is a MAY_OR_MUST borrow+ from p1 to p2, and p1 and p2 are pointers to struct with a field `fld`, then there is a MAY_OR_MUST borrow+ from p1.fld to p2.fld.


## Borrow information

struct Borrow {
    cpath to;
    cpath from;
}

struct BorrowInfo {
    set<Borrow> may_trans_borrows;
    set<Borrow> must_trans_borrows;
}

define bi::may_trans_borrows_to(p)  = [borrow.from for borrow in this.may_trans_borrows  if borrow.to == p]
define bi::must_trans_borrows_to(p) = [borrow.from for borrow in this.must_trans_borrows if borrow.to == p]

define bi::may_trans_borrows_from(p)  = [borrow.to for borrow in this.may_trans_borrows  if borrow.from == p]
define bi::must_trans_borrows_from(p) = [borrow.to for borrow in this.must_trans_borrows if borrow.from == p]



## Borrows on different paths

Consider this snippet:
    
    ```
    int x = 1;
    int y = 2;
    int *p1;
    int *p2;

    if (nondet()) {
        p1 = &x;
        p2 = &y;
    } else {
        p1 = &y;
        p2 = &x;
    }
    *p1 = 3;
    *p2 = 4;
    printf("x=%d, y=%d\n", x, y);
    ```

Statically, at the line "*p1 = 3", p1 may point to x, and p2 may point to x, but there is no execution where both point to x at the same time.  The Rust borrow checker allows this, and so do we.  At the time when the borrows are made, no other borrow exists, so no constraints forbidding the mutable borrow are generated.


## When copying a pointer from src to dest (including PomVarStore, phi, load, store, and copying to "func::return" during a return instruction)

- Let src_base be the first C-path part of src.
- If there is an existing borrow from a C-path rooted at src_base, then
  - Add constraints "!mut(other_dest)" for each Borrow(to=other_dest, from=other_src) where other_src is rooted at src_base.
  - Add a constraint "!mut(dest)".
  - Note: elsewhere, we add constraints "!mut(x) -> !responsible(x)".
- Add Borrow(to=dest, from=src) to BorrowInfo as a MUST borrow.
- Add transitive borrows as described in the section "Transitive borrows".


## GEP for calculating address of struct field

Handle copying the pointer like above, except append ".fieldname" to the source cpath.


## Memory reads

LLVM instruction: `%t = load typ, ptr %p`
Add a constraint that no outstanding borrows rooted at %p are mutable.


## Memory writes

LLVM instruction: `store typ t, ptr %p`
First kill the borrows referring to the memory location about to be overwritten
(see section "Overwriting a memory location holding a borrowed pointer").
If there are any outstanding borrows rooted at %p, then generate a failed constraint `false`.


## Memory frees

LLVM instruction: `call ... @free(ptr %p)`
If there are any outstanding borrows rooted at %p, then generate a failed constraint `false`.


## At beginning of an instruction S

Let incoming = [borrows_at_post[pred] for pred in predecessors(S)];
borrows_at_pre[S].may_trans_borrows = union(bi->may_trans_borrows for bi in incoming);
borrows_at_pre[S].must_trans_borrows = intersection(bi->must_trans_borrows for bi in incoming);

## At function exit point

Check that borrows among formal params and formal "::return" are captured in the provided POM model.
Check that no stack-allocated locals (`alloca`) escape the function (i.e., they shouldn't be borrowed by any cpath rooted at the function parameters or return value).


## When a variable becomes dead

Delete all borrows that refer to the dead variable.
(Note: function args are artificially kept alive thru the end of the function.)


## Overwriting a memory location holding a borrowed pointer

Delete all borrows that refer to the overwritten memory location.
Also delete borrows that refer to MUST location aliases of the memory location.


## Pointer depth

Given a C-path P, we write `ptr_depth(P)` to indicate the pointer depth of P.  For example:
 - If the type of P is `int`, then ptr_depth(P)=0
 - If the type of P is `int*`, then ptr_depth(P)=1
 - If the type of P is `int**`, then ptr_depth(P)=2


## Aliases

A pointer is *eligible to be used for writing* if it is mutable and has no outstanding borrows.

If a pointer p is eligible to be used for writing and no mut constraints are violated, then

    may_ptr_aliases(p) = may_trans_borrows_to(p)
    may_loc_aliases(*p) = [*x for x in may_trans_borrows_to(p)]
    may_loc_aliases(v) = [v] if v is an SSA value

    must_ptr_aliases(p) = must_trans_borrows_to(p)
    must_loc_aliases(*p) = [*x for x in must_trans_borrows_to(p)]
    must_loc_aliases(v) = [v] if v is an SSA value
